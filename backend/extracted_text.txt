1. Présentation Générale
SmartSaha est une application multi-plateforme conçue pour accompagner Madagascar dans s
es activités agricoles. Elle ambitionne de moderniser l'ensemble de la chaîne de valeur agricole : de la gestion des parcelles à la vente des récoltes, en passant par la mise en relation avec les acheteurs, les transporteurs et les organisations agricoles.
Le passage d'une application web unique à une plateforme complète Web + Mobile, soutenue par un backend unifié et une base de données partagée.
1.1 Vision Stratégique
Objectif
Description
Connectivité

Digitalisation
Numériser l'ensemble de la chaîne agricole malgache (production → vente → livraison)
Sécurisation
Garantir des paiements sécurisés locaux (MVola, Orange Money, Airtel) et internationaux (Stripe)
Scalabilité
Architecture cloud évolutive, prête pour la montée en charge

4. Nouveaux Modules à Implémenter
Pour couvrir les besoins de la plateforme mobile et les exigences de la CDC, six nouveaux modules doivent être ajoutés au backend.
4.1 Marketplace
Le Marketplace est transversal Web + Mobile. Les Agricult
eurs publient leurs annonces depuis le Web (avec lien direct vers leur parcelle et récolte), tandis que les Vendeurs purs et les Acheteurs opèrent sur Mobile. Le module gère l'ensemble du cycle de vie d'une annonce : de la publication jusqu'à la livraison.
Types d'Annonces — source_type
source_type
Description
HARVEST
Agriculteur vend sa propre récolte. Lié à une ParcelCrop (parcelle + culture + saison). Disponible sur Web et Mobile.
RESALE
Vendeur revend un produit acheté. Lié à un Post source via source_post. Disponible sur Mobile uniquement.
PURCHASE
Demande d'achat inversée : un acheteur cherche un produit spécifique. Disponible sur Mobile.
Modèles
Modèle
Description
Post
Annonce de vente. Champs clés : culture, quantité (kg), prix/kg, localisation, statut (ACTIVE / SOLD / EXPIRED), source_type (HARVEST / RESALE / PURCHASE), parcel_crop (FK nullable — renseigné si HARVEST), source_post (FK nullable — renseigné si RESALE).
PostImage
Photos associées à une annonce (jusqu'à 5 images). Champ order pour le tri.
Cart
Panier unique par acheteur. Persiste jusqu'à validation ou vidage manuel.
CartItem
Ligne du panier : annonce + quantité demandée + prix snapshotté au moment de l'ajout.
Order
Commande générée lors du checkout. Statuts : PENDING → PAID → CONFIRMED → SHIPPED → DELIVERED / CANCELLED.
Review
Notation croisée après livraison (Acheteur note Vendeur et inversement). Note 1–5 + commentaire.
Favorite
Annonces sauvegardées par un acheteur pour consultation ultérieure.
Fonctionnement de la Revente
Le Revendeur n'est pas un rôle distinct mais un comportement : tout utilisateur ayant reçu une commande avec le statut DELIVERED peut republier ce produit avec son propre prix via le champ source_post.
Règle
Détail
Condition préalable
L'utilisateur doit avoir une Order au statut DELIVERED sur le Post source
Quantité maximale
Quantité revendable ≤ quantité achetée initialement
Prix libre
Le prix de revente est entièrement libre. La marge est calculée et affichée pour la transparence.
Traçabilité
La chaîne de revente est visible : Post A (original) → Post B (revente) → Post C (revente de revente)
Commission réduite
Taux de commission réduit sur les reventes (ex: 3% au lieu de 5%) car une commission a déjà été perçue à l'achat
Panier Multi-Vendeurs
Un acheteur peut ajouter des produits de plusieurs vendeurs dans un seul panier. Lors du checkout, le système génère automatiquement une Order distincte par vendeur, avec une transaction de paiement séparée pour chacun.

Système d'Escrow (Séquestre)
L'argent n'est jamais ve
rsé directement au vendeur au moment du paiement. Il est placé en séquestre sur le compte de la plateforme et libéré uniquement après confirmation de la livraison par l'acheteur. En cas de litige ou d'annulation, l'acheteur est remboursé via le même canal.
Modèles
Modèle
Description
Transaction
Paiement principal. Lie un acheteur, un vendeur et une commande. Statuts : PENDING → SUCCESS / FAILED / REFUNDED. Supporte le paiement partiel (amount_paid + remaining_amount) et le multi-devises (currency + exchange_rate).
PaymentMethod
Moyen de paiement enregistré par un utilisateur (numéro Mobile Money ou carte Stripe). Un moyen par défaut peut être défini.
Escrow
Représente les fonds bloqués en attente de confirmation de livraison. Libéré automatiquement après confirmation ou après délai d'expiration.
Refund
Remboursement déclenché automatiquement (annulation vendeur, expiration 48h, litige) ou manuellement par un admin.
Invoice
Facture PDF générée automatiquement après chaque paiement confirmé. Numérotation automatique (ex: INV-2025-00042).
Commission
Enregistre le taux et le montant perçu par SmartSaha sur chaque transaction. Taux configurable par type (standard / revente).
Dispute
Litige ouvert par un acheteur (non-réception, produit non conforme). Traité par un admin avec décision finale (remboursement ou libération).
4.3 Logistique
Le module Logistique connecte les vendeurs avec des transporteurs pour l'acheminement des commandes. Il inclut la demande de transport, l'estimation des coûts, le suivi en temps réel et la gestion des litiges de livraison.
Modèle
Description
TransportRequest
Demande de transport liée à une commande. Contient : adresse enlèvement, adresse livraison, coût estimé, statut (PENDING → ACCEPTED → IN_TRANSIT → DELIVERED).
TransportTracking
Historique JSON des mises à jour de position et de statut du transporteur.
TransportDispute
Litige de livraison (colis endommagé, retard, non-livraison). Traité par un admin.

4.5 KYC — Vérification d'Identité
Le module KYC (Know Your Customer) permet de valider l'identité des producteurs et vendeurs avant qu'ils puissent publier des annonces ou recevoir des paiements. L'administrateur examine les documents soumis et approuve ou rejette la demande.
Champ
Description
doc_type
Type de document : CNI, Passeport, Registre du commerce, Carte professionnelle
file_url
URL sécurisée du document uploadé (stockage cloud)
status
État du dossier : PENDING → APPROVED / REJECTED
reviewed_by
Admin ayant traité le dossier
reviewed_at
Date et heure de la décision
4.6 Mobile Auth — Vérification par SMS
L'authentification mobile s'appuie sur une vérification OTP (One-Time Password) par SMS en plus de l'email. Le code est valable 10 minutes et stocké temporairement en cache Django (non persisté en base de données).
Étape
Endpoint
1. Demande d'OTP
POST /api/mobile/auth/send-otp/  →  Envoie un SMS avec code à 6 chiffres
2. Vérification OTP
POST /api/mobile/auth/verify-otp/  →  Valide le code, active le compte
3. Connexion
POST /api/mobile/auth/login/  →  Retourne access token JWT + refresh token
Fournisseur SMS recommandé pour Madagascar : Africa's Talking (support MG, API simple) ou OrangeSMS Madagascar pour une intégration locale.
